# 測定モデルと rank に関する覚書

作成: 2026-04-24  
経緯: Oracle の rank=4 の解釈について議論し、認識を整理した。

---

## 1. PSF なし・ブロック仮定なしの場合（最も素朴な見方）

各スナップショットピクセル (y,x) の観測方程式：

```
snapshot[y,x] = (1/N) × Σ_s mask[s,y,x] × volume[s,y,x]
```

- 観測：1 個（snapshot[y,x] というスカラー）
- 未知数：11 個（volume[0,y,x], volume[1,y,x], ..., volume[10,y,x]）
- **1 方程式・11 未知数 → rank=1 per pixel → 完全に underdetermined**

重要：隣接ピクセル (y,x) と (y,x+1) の未知数は **完全に別物**。方程式に共有する未知数はない。

---

## 2. Oracle の「ブロック均一」仮定

Oracle（`my-program/oracle_reconstruction.py`）では以下の近似を導入：

**2×2 ブロック内のすべてのピクセルが同じ深度プロファイルを持つと仮定する**

```
volume[s, 0, 0] ≈ volume[s, 0, 1] ≈ volume[s, 1, 0] ≈ volume[s, 1, 1] = V_s
```

これにより 2×2 ブロック内の 4 方程式が**同じ 11 個の未知数 V_0...V_10 を共有**する：

```
y₀₀ = (1/N) × Σ_s A[0,s] × V_s
y₀₁ = (1/N) × Σ_s A[1,s] × V_s
y₁₀ = (1/N) × Σ_s A[2,s] × V_s
y₁₁ = (1/N) × Σ_s A[3,s] × V_s
```

ここで A[i,s] = mask[s, i//2, i%2]（ピクセル位置 i の スライス s のマスク値）

- 観測：4 個
- 未知数：11 個（共有）
- **rank=4 → 4 次元の情報しか取れない**

**rank=4 はこの仮定の下での話であり、ブロック均一仮定がない実際のシステムには直接適用されない。**

---

## 3. 実際の DIP システム（PSF あり）

MaskedImagingModel の forward model：

```
snapshot[y,x] = (1/N) × Σ_s mask[s,y,x] × (PSF_s ⊗ volume_s)[y,x]
```

PSF の畳み込み `PSF_s ⊗ volume_s` により、snapshot[y,x] が**周囲の広い空間範囲のボリューム値**に依存する。

- 隣接ピクセルの方程式が間接的に未知数を共有 → 全体として豊かな拘束
- 全画像: 16,384 観測（snapshot pixels）で 180,224 未知数（volume voxels）を拘束
- PSF の空間結合 + DIP の U-Net 事前分布（null 空間を滑らかさで埋める）により 37 dB を達成

---

## 4. 各モデルの比較

| モデル | 観測数/ブロック | 未知数/ブロック | rank | 備考 |
|---|---|---|---|---|
| PSF なし・仮定なし | 4 | 4×11=44 | 1 per pixel | 最悪。各ピクセル独立 |
| Oracle（ブロック均一） | 4 | 11 | 4 | 仮定により共有。rank=4 はここでの話 |
| 実 DIP（PSF あり） | 16,384（全画像） | 180,224 | PSF で増加 | 空間結合があるため Oracle より豊か |

---

## 5. 「rank=4 がボトルネック」という言い方の正確な意味

「rank=4」は Oracle（PSF なし・ブロック均一）という**近似モデルの下でのボトルネック**。

実際の DIP は PSF による空間結合があり rank はより高い。ただし：
- 1 スナップショット・2×2 タイリングという構造上、情報量には根本的な制約がある
- DIP は事前分布で null 空間を補完しており、その補完が 24 dB 分を担っている
- パターン最適化（Phase B）は「DIP の事前分布が埋めやすい null 空間を作る」方向に効く可能性がある

**「2×2 ブロックでは rank が 4 を超えられない」は Oracle モデルの話であって、  
DIP + PSF の実システムでこの rank がそのまま上限になるわけではない。**

---

## 6. Phase B への含意

- Oracle の rank=4 は「ブロック均一近似下での情報上限」
- 実システムでは PSF 結合があるため状況は Oracle より良い
- Phase B で 2×2 タイリングのまま pattern を最適化しても「Oracle での rank」は変わらない
- ただし「DIP が扱いやすいパターン」に最適化することで実際の PSNR を改善できる可能性はある
- より根本的な改善には：ブロックサイズ変更（4×4 → 16 観測）・複数スナップショット・非均一パターン等の構造変更が必要
